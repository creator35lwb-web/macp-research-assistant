### Anti-Hallucination Protocol (Critical Credibility Fix)

**Root cause:** Validation of arXiv:2412.12004 ("The Open Source Advantage in LLMs") exposed a hallucination bug. PyMuPDF only extracted the title/author page (PDF formatting issue). `sections[0]["content"]` contained only the paper title (~50 chars). Gemini received that as "abstract + introduction" text and generated 5 specific key findings with 0 actual citations — pure fabrication.

**Three-layer fix deployed:**

1. **paper_fetcher.py** — Quality gate + HTML fallback
   - `MIN_BODY_CHARS = 500` threshold constant
   - `check_extraction_quality(extracted) → dict` — counts total chars and body sections, returns `is_sufficient` flag
   - `fetch_arxiv_html(arxiv_id) → Optional[dict]` — fetches `arxiv.org/html/{id}`, strips with stdlib `HTMLParser` (no new deps), calls existing `_split_into_sections()`

2. **webmcp.py** — Quality gate in `/analyze-deep` endpoint
   - After PDF extraction: calls `check_extraction_quality()`
   - If `is_sufficient=False`: tries `fetch_arxiv_html()` as fallback
   - Logs warning/info at each stage
   - Passes `extraction_source` ("pdf" | "html_fallback") to LLM layer and includes in response

3. **llm_providers.py** — Anti-hallucination rules in all 3 pass prompts
   - CRITICAL RULES block added to DEEP_PASS1/2/3_PROMPT
   - LLM must set `extraction_warning: "INSUFFICIENT TEXT..."` if <100 words
   - Must not extrapolate from title alone; fallback to "Not found in extracted text"
   - `extraction_warning` field added to all 3 pass JSON schemas
   - `analyze_paper_deep()` now accepts `extraction_source` param
   - `_meta` now includes `extraction_source` + `extraction_warnings: [list]`

4. **generateMarkdown.ts + types.ts** — Frontend warning surface
   - `DeepAnalysisResult._meta` type extended with `extraction_source?` and `extraction_warnings?`
   - Extraction warning callout rendered in Markdown export if warnings present
